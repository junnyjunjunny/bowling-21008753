# Plan: 10번째 프레임에서 스페어/스트라이크로 인한 추가 투구

## 목표 (RED)

10번째 프레임이 스페어 또는 스트라이크로 끝나 추가 투구가 발생하는 경우, 그 추가 투구가 보너스로 정확히 반영되어 최종 점수가 올바르게 계산되는지 검증하는 테스트 두 개를 작성한다.

## 작성할 테스트

파일: `tests/test_game.py`

```python
def test_tenth_frame_spare_gets_one_bonus_roll():
    game = Game()

    for _ in range(18):  # 1~9프레임: 전부 거터
        game.roll(0)

    game.roll(5)
    game.roll(5)  # 10프레임: 스페어
    game.roll(1)  # 보너스 투구 1회

    assert game.score() == 11


def test_tenth_frame_strike_gets_two_bonus_rolls():
    game = Game()

    for _ in range(18):  # 1~9프레임: 전부 거터
        game.roll(0)

    game.roll(10)  # 10프레임: 스트라이크
    game.roll(5)
    game.roll(3)   # 보너스 투구 2회

    assert game.score() == 18
```

- 첫 번째 테스트: 1~9프레임 0점, 10프레임 = 5+5+1(보너스) = 11. 합계 11.
- 두 번째 테스트: 1~9프레임 0점, 10프레임 = 10+5+3(보너스) = 18. 합계 18.
- 사전 확인 결과, 현재 구현은 `for _ in range(10)` 루프가 정확히 10번만 반복되고 종료되므로, 10프레임 이후에 남은 보너스 투구를 "11번째 프레임"으로 잘못 계산하려는 시도 자체가 없다. 스페어/스트라이크 분기 로직이 이미 10프레임에도 동일하게 적용되어, **두 테스트 모두 이미 정확한 값을 반환한다**.
- 즉 이번 사이클도 회귀 테스트 성격이다 — 새 로직을 추가하는 것이 아니라, 10번째 프레임이라는 경계 케이스에서도 기존 스페어/스트라이크 계산이 깨지지 않음을 명시적으로 문서화하는 것이 목적이다.

## 예상되는 최소 구현 범위 (GREEN에서 진행 예정)

- `src/game.py` 변경 불필요할 가능성이 높다.
- 두 테스트 모두 실제로 통과하면 그대로 GREEN 확인만 하고 넘어간다.
- 예상과 달리 실패한다면 (예: 인덱스 초과, 보너스 오계산 등), 원인을 확인한 뒤 이 Plan을 다시 검토한다.

## 회귀 확인

- 기존 5개 테스트가 새 테스트 추가 후에도 계속 통과하는지 함께 확인한다.

## 이번 사이클 이후 남은 항목

- 이 두 테스트까지 통과하면, 요구사항에 명시된 볼링 점수 계산의 핵심 케이스(거터/일반/스페어/스트라이크/퍼펙트 게임/10프레임 보너스)를 모두 다룬 것이 된다.
- 입력 유효성 검증은 카타 범위에서 명시적으로 제외되었으므로 별도 사이클을 진행하지 않는다.
